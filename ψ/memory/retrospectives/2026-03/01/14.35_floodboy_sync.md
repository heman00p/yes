## Honest Feedback
1. **Directory Mismanagement Friction**: Moving files between directories during a git session caused confusion. I should have staged the move more explicitly instead of relying on a multi-command shell string that failed on certain OS flags (`mv -rf` vs `Move-Item`).
2. **Branch Name Ambiguity**: The mismatch between `main` and `master` (remote default) was a silent blocker for a moment. It's a classic friction point that requires early verification of `git remote show origin`.
3. **Wait Time for Deployment**: The delay between pushing to GitHub and the site going live created a "doubt loop" where we weren't sure if the deploy actually happened. Using `gh run list` earlier would have provided better signal than just checking `git log`.

## Lessons Learned
- **The Divisor Rule (x10000)**: Never assume standard decimal places for IoT data on-chain. Always verify the `unit` string and implement explicit scaling (e.g., dividing by 10000 for meters).
- **Deployment Verification**: Always use `gh run list` to monitor the actual progress of CI/CD workflows instead of just relying on the git push confirmation.
- **Root-Level Accessibility**: For simple GitHub Pages deployments, keeping major entry points at the root (or explicitly in a subfolder mentioned in the build step) prevents most 404 routing issues.
